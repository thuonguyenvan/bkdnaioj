# Phase 5: Queue Protocol (Redis Streams)

**Status:** 🆕 New
**Refs:** phases 02/03/04

---

## 1. Why Redis Streams

- Cross-language (Go producer, Python consumer) without Celery/asynq incompatibility
- Persistent log + consumer groups + XPENDING reclaim → at-least-once delivery
- Built into Redis 7 (already in stack)
- Simple JSON envelope; no client library lock-in

---

## 2. Stream Topology

| Stream | Producer | Consumer Group | Consumers |
|---|---|---|---|
| `jobs:validate` | Go (on submission upload) | `cg:validate` | Python validators |
| `jobs:judge`    | Python validator | `cg:judge` | Python judges |
| `jobs:score`    | Python judge | `cg:score` | Python scorers |
| `jobs:rejudge`  | Go (admin action) | `cg:rejudge` | Python rejudgers |
| `jobs:dlq`      | Sweeper (Go or Python) | `cg:dlq-monitor` | Go (auto-ticket creator) |
| `jobs:results`  | Python scorer (success or failure) | `cg:ws-bridge` | Go (WS broadcaster + leaderboard recompute) |

## 3. Envelope Schema

```json
{
  "job_id": "01HXYZ...",          // ULID generated by producer
  "submission_id": "uuid",
  "job_type": "validate|judge|score|rejudge",
  "priority": 5,
  "attempt": 1,
  "enqueued_at": "2026-04-24T12:34:56Z",
  "trace_id": "uuid",
  "judge_result": { ... }          // only set on jobs:score after judge stage
}
```

Stored as Redis stream entry fields (one field `payload` containing JSON string, KISS).

## 4. Producer (Go)

```go
// internal/queue/producer.go
type JobEnvelope struct {
    JobID        string          `json:"job_id"`
    SubmissionID uuid.UUID       `json:"submission_id"`
    JobType      string          `json:"job_type"`
    Priority     int             `json:"priority"`
    Attempt      int             `json:"attempt"`
    EnqueuedAt   time.Time       `json:"enqueued_at"`
    TraceID      uuid.UUID       `json:"trace_id"`
    JudgeResult  json.RawMessage `json:"judge_result,omitempty"`
}

func (p *Producer) Enqueue(ctx context.Context, stream string, env JobEnvelope) error {
    payload, _ := json.Marshal(env)
    return p.rdb.XAdd(ctx, &redis.XAddArgs{
        Stream: stream,
        MaxLen: 100_000,
        Approx: true,
        Values: map[string]any{"payload": string(payload)},
    }).Err()
}
```

Called from submission handler after row + files persisted:
```go
producer.Enqueue(ctx, "jobs:validate", JobEnvelope{...})
```

## 5. Consumer (Python)

```python
# workers/app/queue.py
def consume(stream: str, group: str, consumer: str, handler):
    rdb.xgroup_create(stream, group, id="$", mkstream=True)  # idempotent
    while True:
        resp = rdb.xreadgroup(group, consumer,
                              streams={stream: ">"},
                              count=BATCH, block=5000)
        for _stream, msgs in resp or []:
            for msg_id, fields in msgs:
                env = json.loads(fields[b"payload"])
                try:
                    handler(env)
                    rdb.xack(stream, group, msg_id)
                except RetryableError:
                    pass  # leave un-acked → reclaimed
                except FatalError:
                    rdb.xack(stream, group, msg_id)
                    rdb.xadd("jobs:dlq", {"payload": json.dumps({**env, "fatal_reason": str(e)})})
```

## 6. Reclaim Sweeper

Background goroutine in Go (and/or Python script) running periodically:

```
For each (stream, group):
  pending = XPENDING stream group IDLE 60000 - + 100 *
  for each pending msg with attempt < max_attempts:
      XCLAIM stream group consumer min-idle-time msg-id
      increment attempt
  for each pending msg with attempt >= max_attempts:
      XADD jobs:dlq payload
      XACK to remove from PEL
      mark submission failed + auto-create ticket
```

Run every 30s. Owned by Go API process (single instance with leader-election via Redis SETNX lock to avoid duplicate sweeps).

---

## 7. Retry Policy

| Stage | Max attempts | Backoff |
|---|---|---|
| validate | 3 | 0s, 5s, 30s |
| judge | 3 | 0s, 30s, 120s |
| score | 5 | 0s, 1s, 5s, 30s, 60s |
| rejudge | 3 | same as judge |

Backoff handled by sweeper using `attempt` field — consumer just retries on each delivery; sweeper enforces idle window between claims.

---

## 8. DLQ Handling

- `jobs:dlq` stream is consumed by Go DLQ-monitor consumer
- For each DLQ message:
  - Update `submissions.status='failed'` (idempotent), `error_message=fatal_reason`
  - Insert `tickets(category='judge'|'system', auto-created, assigned_to=jury-rotation)`
  - Broadcast WS `submission.failed` to entry members
  - XACK

## 9. Result Stream → WS Bridge (Go)

```go
// internal/queue/consumer.go
func (b *Bridge) Run(ctx context.Context) error {
    rdb.XGroupCreateMkStream(ctx, "jobs:results", "cg:ws-bridge", "$")
    for {
        msgs, _ := rdb.XReadGroup(ctx, &redis.XReadGroupArgs{
            Group: "cg:ws-bridge", Consumer: hostname,
            Streams: []string{"jobs:results", ">"},
            Count: 50, Block: 5*time.Second,
        }).Result()
        for _, m := range msgs[0].Messages {
            var r ResultEnvelope
            json.Unmarshal([]byte(m.Values["payload"].(string)), &r)
            // 1) Recompute task-phase board for this entry/phase
            srv.RecomputeTaskPhaseBoard(ctx, r.SubmissionID)
            // 2) Recompute contest-phase board
            srv.RecomputeContestPhaseBoard(ctx, r.SubmissionID)
            // 3) Broadcast WS
            wsHub.PublishSubmission(r.SubmissionID, r)
            wsHub.PublishLeaderboardUpdated(r.PhaseID, r.ContestPhaseDefID)
            rdb.XAck(ctx, "jobs:results", "cg:ws-bridge", m.ID)
        }
    }
}
```

## 10. Single-Writer Rule (recap, enforced via discipline + code review)

| Stream | Allowed DB writes |
|---|---|
| Go `Enqueue` | submissions INSERT only |
| Python validator | submissions UPDATE: status, validation_result; eval_jobs INSERT/UPDATE |
| Python judge | submissions UPDATE: status, error_message; eval_jobs INSERT/UPDATE |
| Python scorer | submissions UPDATE: raw_score, display_score, score_payload, evaluated_at, status |
| Go `Bridge` | leaderboard UPSERT, NOT submissions |

Code review checklist enforces no overlap.

## 12. Failure Modes

| Mode | Detection | Recovery |
|---|---|---|
| Worker crash mid-job | XPENDING idle | Sweeper XCLAIM |
| Bad envelope JSON | Worker parse error | XACK + DLQ |
| Postgres deadlock on score update | Worker exception | Retry up to max → DLQ |
| Redis down | Producer error → 5xx to client | Client retries; submission stays `uploaded` until next enqueue (manual retry endpoint) |
| Bridge crash | Pending entries in jobs:results | Auto-resume via consumer group |

## 14. Success Criteria

1. At-least-once delivery verified (kill worker mid-job → message reclaimed)
2. DLQ catches poison messages without blocking pipeline
3. Bridge keeps WS in sync (< 1s lag from XADD to broadcast)
4. Single-writer rule never violated (verified via concurrent stress test)
5. Trace IDs flow across Go ↔ Python boundary

## 15. Unresolved Questions

1. **Backpressure:** if `jobs:judge` lags, should producer slow down or queue infinitely (with MAXLEN guard)? Currently MAXLEN=100k approx — acceptable for V1.
2. **Multi-region:** scope V1 is single-region; Streams not replicated cross-region. OK for thesis.
3. **Sweeper leader election:** Redis SETNX with TTL vs full Raft? V1 → SETNX is enough.
4. **Idempotency on `jobs:results`:** rare double-delivery → ensure leaderboard upsert is idempotent (already is via UNIQUE constraint).
5. **Submission retry endpoint** when Redis down at submit time — manual `POST /submissions/{id}/enqueue` (admin) needed?
